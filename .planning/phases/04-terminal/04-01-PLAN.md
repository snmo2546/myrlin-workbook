<objective>
Build the hybrid WebView terminal renderer with xterm.js, the postMessage bridge protocol, WebSocket connection (inside the WebView), theme injection, and activity indicator header.

Purpose: This is the foundation for all terminal interaction. The WebView acts as a dumb rendering surface for xterm.js output while React Native controls all user interaction. The WebSocket for PTY data runs inside the WebView for performance (no RN bridge overhead for high-volume terminal output).

Output: A navigable terminal screen that renders real-time PTY output with correct theme colors, accessible from the session detail screen.
</objective>

From src/web/pty-manager.js (WebSocket protocol):
- Client connects to `ws://{host}/ws/terminal?token={token}&sessionId={id}`
- Client sends: `{ type: 'resize', cols, rows }` or `{ type: 'input', data: string }`
- Server sends: raw text (PTY output) OR `JSON.stringify({ type: 'exit', exitCode })` for session exit
- On connect, server replays scrollback buffer, then streams live output

Create three foundational files:

**1. mobile/types/terminal.ts** - Bridge protocol type definitions (from research doc):
- `ToWebView` union type with messages: write, resize, setTheme, clear, connect, disconnect, dispose, getSelectedText, getVisibleText, getScrollback, selectAll, scrollToBottom, focus, blur
- `FromWebView` union type with messages: ready, selectedText, visibleText, scrollback, activity, bell, titleChange, dimensions, disconnected, exit
- `XtermTheme` interface matching xterm.js ITheme (22 color properties: background, foreground, cursor, cursorAccent, selectionBackground, selectionForeground, black/red/green/yellow/blue/magenta/cyan/white, and bright variants)
- `myrlinToXtermTheme(theme: MyrlinTheme): XtermTheme` converter function. Map: base->background, text->foreground, rosewater->cursor, base->cursorAccent, accent+'40'->selectionBackground, text->selectionForeground, surface1->black, red->red, green->green, yellow->yellow, blue->blue, mauve->magenta, teal->cyan, subtext1->white, surface2->brightBlack, subtext0->brightWhite. Bright colors match their normal counterparts.

**2. mobile/assets/terminal.html** - Self-contained HTML with inlined xterm.js:
- CRITICAL: Bundle xterm.js v5 and xterm-addon-fit v0.8 INLINE in the HTML (not CDN). Download the minified sources and embed them in `<script>` tags. This avoids CDN dependency for LAN-only setups.
- Viewport meta: `width=device-width, initial-scale=1, maximum-scale=1, user-scalable=no`
- CSS: `* { margin: 0; padding: 0; box-sizing: border-box; } html, body, #terminal { width: 100%; height: 100%; overflow: hidden; background: #1e1e2e; }`
- Terminal config: `cursorBlink: false, disableStdin: true, fontSize: 13, fontFamily: "'JetBrains Mono', 'Courier New', monospace", scrollback: 5000, allowTransparency: false`
- `window.handleRNMessage(msg)` handler supporting all ToWebView message types
- `postToRN(msg)` using `window.ReactNativeWebView.postMessage(JSON.stringify(msg))`
- WebSocket connection via `connect` message (receives wsUrl, token, sessionId). Build full URL: `wsUrl + '?token=' + encodeURIComponent(token) + '&sessionId=' + sessionId`
- WS onopen: send resize with current term.cols/rows
- WS onmessage: try JSON.parse for `{ type: 'exit' }` messages, otherwise `term.write(e.data)` for raw PTY output
- WS onclose: `postToRN({ type: 'disconnected' })`
- Double-rAF fit pattern on load: `requestAnimationFrame(() => requestAnimationFrame(() => { fitAddon.fit(); postToRN({ type: 'ready' }); postToRN({ type: 'dimensions', cols: term.cols, rows: term.rows }); }))`
- ResizeObserver on body to re-fit terminal on WebView resize
- getSelectedText, getVisibleText, getScrollback handlers that extract text from xterm.js buffer and postToRN
- dispose handler: close WS, call term.dispose()

**3. mobile/components/terminal/useTerminalBridge.ts** - React hook for bridge communication:
- `useTerminalBridge()` returns `{ webViewRef, sendToWebView, handleWebViewMessage, isReady, dimensions, isConnected }`
- `webViewRef` is `useRef<WebView>(null)`
- `sendToWebView(msg: ToWebView)` injects JS: `window.handleRNMessage(${JSON.stringify(msg)}); true;`
- `handleWebViewMessage` callback parses `event.nativeEvent.data` as `FromWebView`, routes by type
- Track `isReady` (set true on 'ready' message), `dimensions` (from 'dimensions' message), `isConnected` (true after ready, false on 'disconnected')
- Accept callbacks: `onActivity?: (kind: string, detail: string) => void`, `onExit?: (code: number) => void`, `onReady?: () => void`, `onDimensions?: (cols: number, rows: number) => void`, `onText?: (type: string, text: string) => void`
- These callbacks are stored in refs to avoid stale closures
